# Phase 1 Data Model: Postgres Migration

**Feature**: 009-postgres-migration  
**Date**: 2026-06-02  
**Scope**: Postgres-side schema only. Per FR-008 the **ORM models, Pydantic
schemas, and request/response shapes do not change** — they are reused from
[backend/src/momdiary/models/orm.py](backend/src/momdiary/models/orm.py).
This document describes the **on-disk Postgres schema** that the new
`0004_postgres_baseline.py` revision produces and the **one new entity**
(`chat_sessions`) introduced by this feature.

---

## A. Existing relational tables — reused, recreated natively

[0002_users_babies.py](backend/alembic/versions/0002_users_babies.py), and
[0003_clerk_users.py](backend/alembic/versions/0003_clerk_users.py) are
recreated in `0004_postgres_baseline.py` with the following
Postgres-native adjustments. **No column semantics change.**

| Table                | Source revision(s)        | Postgres adjustments                                                       |
| -------------------- | ------------------------- | -------------------------------------------------------------------------- |
| `users`              | 0001, 0002, 0003          | `email` unique constraint becomes plain `UNIQUE` (no `COLLATE NOCASE`); see Decision 9 |
| `user_sessions`      | 0002                      | None beyond dialect normalisation                                          |
| `babies`             | 0001, 0002                | None                                                                       |
| `feeds`              | 0001                      | None                                                                       |
| `sleeps`             | 0001                      | None                                                                       |
| `poops`              | 0001                      | None                                                                       |
| `appointments`       | 0001                      | None                                                                       |
| `appointment_notes`  | 0001                      | None                                                                       |
| `agent_interactions` | 0001                      | Any `JSON` columns declared as `JSON` use `JSONB` in Postgres (lossless)   |
| `settings`           | 0001                      | None                                                                       |

**Ownership / scoping (preserved verbatim)**:

- Every diary row carries a `baby_id NOT NULL` FK (added in feature 006).
- `babies.user_id` FK ties each baby to its owning caregiver.
- Per-user scoping is enforced at the application layer in the existing
  service modules and tested by the existing contract suite. FR-007 / SC-002
  validate that this continues to hold post-migration.

**Foreign-key behaviour**: identical to today (`ON DELETE` rules unchanged).
Postgres enforces FKs immediately, same observable behaviour as SQLite's
`foreign_keys=ON` PRAGMA (which the SQLite engine setter enabled).

## B. New entity: `chat_sessions`

The only schema *addition* in this feature.

### Purpose

Replaces the in-process `InMemorySessionStore` dict so that any backend
worker or instance can read and resume any unexpired chat session
(satisfies FR-002 / SC-002).

### Definition

```sql
CREATE TABLE chat_sessions (
    session_id   VARCHAR(64)   NOT NULL PRIMARY KEY,
    user_id      VARCHAR(64)   NOT NULL,
    baby_id      INTEGER       NOT NULL,
    turns        JSONB         NOT NULL DEFAULT '[]'::jsonb,
    updated_at   TIMESTAMPTZ   NOT NULL DEFAULT NOW()
);
CREATE INDEX ix_chat_sessions_user_baby_updated
    ON chat_sessions (user_id, baby_id, updated_at DESC);
CREATE INDEX ix_chat_sessions_updated_at
    ON chat_sessions (updated_at);
```

### Fields

| Field        | Type          | Notes                                                                                                       |
| ------------ | ------------- | ----------------------------------------------------------------------------------------------------------- |
| `session_id` | `VARCHAR(64)` | Opaque client/server identifier (existing — generated by current session-store code).                       |
| `user_id`    | `VARCHAR(64)` | Owning caregiver. Matches the type used by the existing `ChatSession.user_id` dataclass field.              |
| `baby_id`    | `INTEGER`     | Baby the session is about. Stored alongside `user_id` so eviction and per-user purge are single-table ops. |
| `turns`      | `JSONB`       | Ordered list of `ChatTurn` objects (caregiver + assistant pairs). FIFO-trimmed and per-message byte-capped *in Python* before write, using the same helpers the in-memory store already uses. Default `'[]'::jsonb` so an empty session inserts cleanly. |
| `updated_at` | `TIMESTAMPTZ` | Last write time. Drives TTL eviction (FR-012) and the recent-sessions ordering on the composite index. |

### Validation rules

| Rule                                                                       | Where enforced                                          |
| -------------------------------------------------------------------------- | ------------------------------------------------------- |
| `session_id` is the upsert key; one row per session.                       | `PRIMARY KEY` constraint.                               |
| Turn count never exceeds `MOMDIARY_SESSION_MAX_TURNS` (default 50).        | Python FIFO trim before upsert (reused helper).         |
| Per-message bytes never exceed `MOMDIARY_SESSION_MESSAGE_MAX_BYTES` (4096).| Python byte-cap before upsert (reused helper).          |
| Idle sessions older than `MOMDIARY_SESSION_TTL_SECONDS` (86_400 = 24h) are deleted. | Background sweeper (Decision 6) — single advisory-lock holder runs `DELETE ... WHERE updated_at < NOW() - make_interval(secs => :ttl)`. |
| TLS required on every connection.                                          | `?ssl=require` in URL; Postgres server-side enforcement.|

### State transitions

```
[no row]
   |
   |  first turn appended for a new session_id
   v
[row exists, turns = [t1]]
   |
   |  subsequent appends within the TTL window
   v
[row exists, turns = [t1..tN]  (N capped at MAX_TURNS via FIFO)]
   |
   |  no writes for TTL seconds
   v
[deleted by background sweeper]
```

A purge (e.g. account deletion path that already exists) deletes all rows
matching the user's `user_id` in a single statement.

### Relationships

- `chat_sessions.user_id` references a Clerk user id (same id space as
  `users.clerk_user_id` from feature 008). It is **not** modelled as a
  database FK — sessions can outlive a transient user-table consistency
  blip, and current code does not FK the in-memory equivalent. App-layer
  scoping (FR-007) is unchanged.
- `chat_sessions.baby_id` similarly is **not** a database FK. Per-user
  scoping enforced by the existing service layer ensures a caller can
  only see sessions for babies they own.

(Rationale for not adding FKs: the in-memory store had no FKs either;
adding them now would change observable behaviour on user/baby deletion,
which is out of scope for this feature.)

### Reused Python types

- `ChatSession` dataclass (existing, in
  [backend/src/momdiary/agents/session_store.py](backend/src/momdiary/agents/session_store.py))
  — unchanged.
- `ChatTurn` dataclass (existing, same file) — unchanged.
- Serialisation to/from JSONB uses the standard JSON encoder applied to the
  `ChatTurn` list; no new schema or codec is introduced.

## D. Alembic revision graph after this feature

```
0001_initial          (SQLite-only, historical)
   └── 0002_users_babies   (SQLite-only, historical)
          └── 0003_clerk_users   (SQLite-only, historical)

0004_postgres_baseline   (down_revision = None)
   - creates: every table from A
   - creates: chat_sessions from B
   - is the only revision a fresh Postgres database ever sees
```

A fresh `alembic upgrade head` against an empty Postgres yields exactly the
schema documented above.
